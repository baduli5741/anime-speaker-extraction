# 실험 로그

`cross-lingual-voice-cloning` repo(GPT-SoVITS 학습/더빙 담당)의 16장에서 다루던 데이터 추출 실험을 이 repo로 이관·통합. 이관 이전 배경은 그쪽 로그 16-8~16-14 참고.

## 1. 왜 재추출이 필요했나

GPT-SoVITS v2Pro로 22캐릭터 학습 후 QC 중 **페른(フェルン) 목소리만 이상함** 발견. 진단: 학습샘플 6개 중 5개 불량(83% 오염) — 페른이 항상 프리렌과 붙어있는 캐릭터라 화자섞임이 몰림. 힘멜·슈타르크·자인도 스팟체크 결과 짤림·화자겹침·시대다른 목소리 혼재 등 발견. **기존 추출기(자막 타이밍 그대로 슬라이스 + 임계값 기반 이름매칭)가 근본적으로 부정확**하다는 결론 → 재설계 결정.

## 2. 기존 방식의 문제 (build_global_centroids.py 등, legacy/ 참고)

임계값(0.4) 코사인 매칭: 애매하면 `_unknown`행(1047개 유실) 또는 비슷한 목소리끼리(페른↔프리렌) 오매칭. "임계값 매칭"이라는 접근 자체의 근본 한계.

## 3. GitHub 유사 프로젝트 서베이

- **TTSizer** (MelBandRoformer+CTC정렬+Wespeaker+Gemini다이어) — 애니 데이터셋 실적(AnimeVox) 있음, CLI
- **AnimeSpeech** (자막기반, 로컬, 임베딩+KNN+사람라벨) — 우리 요구사항(로컬·자막·일본어)과 유사
- **speech-dataset-generator** (pyannote)

## 4. Path A: TTSizer 시도 (클라우드, 결국 미채택)

셋업 중 버그 다수 수정:
- Gemini 파일업로드 `.flac` mime 미인식 → `config={"mime_type":"audio/flac"}`
- `gemini-3.5-flash` 503 과부하 → `gemini-2.5-flash`로. `gemini-2.5-pro`는 무료키에서 429(limit:0)
- 다이어라이제이션 프롬프트가 텍스트를 영어로 번역 → 일본어 강제 프롬프트로 수정
- CTC 정렬 실패 원인 = `language_code`가 `"ja"`(라이브러리는 ISO-639-3 `"jpn"` 요구) + 일본어는 공백없어 `min_words_per_segment`가 2→1 필요. perl/uroman 자체는 정상이었음(오진단 정정)
- `main.py`에 diarizer 호출이 **주석처리**돼 있어 원래 전혀 안 돌고 있었음

결과: 다이어 304세그, 정렬 239클립(에러0), outlier검출로 오염 재발 방지 성공. **단 클립이 매우 짧음(0.2~1.9초)** — 짧은 클립은 페른 오염사고와 같은 패턴(학습 불안정 위험). 클라우드(Gemini) 의존이라 완전로컬 요구사항에도 안 맞음 → **미채택**.

## 5. Path B: 자막기반 로컬 (채택)

### 5.1 v1 — CTC 강제정렬 기반 클립 생성
자막 텍스트를 CTC로 오디오에 정렬(324/324 성공, 우회: 서버 perl 손상 → `FindBin.pm` 스텁+`PERL5LIB`). 경계는 tight했으나 두 문제 발견:
1. 일부 클립 말중간 잘림
2. 일부 클립에 2화자 혼입(자막 종료시각이 다음 화자 시작과 겹침)

### 5.2 v3 — sub-window 화자전환 필터 (완전 실패)
클립을 앞/뒤 40%로 쪼개 임베딩 비교 → 화자전환 의심시 폐기. **실측(calibrate_thresh.py)으로 검증한 결과 완전 실패**:
```
SAME-speaker  n=87  mean=0.258  (같은 화자인데도 낮음)
DIFF-speaker  n=40  mean=0.163
분리마진: -0.423 (겹침 심함, 못 씀)
```
원인: 클립이 1~3초라 반쪼개면 0.35~0.8초 조각 → 화자정보보다 발음내용에 좌우됨. **결론: 짧은클립 반쪼개기 방식 자체가 원천적으로 안 됨.** 탐지보다 예방(애초에 잘 자르기)으로 방향 전환.

### 5.3 오염 원인 역추적 (아이젠+프리렌 cluster_02 사례)
아이젠 대사 원본SRT `318.15~320.32`(표시 2.17초)인데 실제발화는 더 짧음. **자막 종료시각은 가독성 위해 늘려짜여진 값**(업계표준 0.5~2초 연장 — subtitle timing 표준 문서로 확인)이라 신뢰 불가. 빠른 대사교환 구간에서 이 여유가 다음화자 시작과 겹쳐 CTC 검색창이 오염.

### 5.4 v4→v5 — 이웃경계 하드클램프
v4(1차): `kept`(필터후) 리스트로 클램프 → **실패**. 문제의 다음줄이 화자태그 2개라 이미 필터링에서 제외돼 있어서 클램프가 못 봄.
v5(2차): **원본 SRT 전체(미필터, `segs`)** 기준 클램프로 교체 → 해결 확인(아이젠 케이스 320.833→320.403).

### 5.5 v6 — 가드패딩 순서버그 수정
v5에 순서버그: 클램프 후 가드패딩(+0.05s)을 나중에 더해서 클램프효과 상쇄(하이터 웃음이 프리렌 클립 끝에 유입, 페른 대사도 여유0으로 접촉). **가드패딩을 클램프보다 먼저** 적용하도록 순서교체 → 두 독립사례 모두 해결.

### 5.6 v6 잔여 문제 (미해결)
청취검증 중 3건의 꼬리누출 추가발견, 두 하위원인:
- **하위원인 A**(힘멜·하이터, 여유 0.046~0.05초): 고정 안전마진이 너무 좁음
- **하위원인 B**(아이젠, 자막상 여유 0.235초인데도 유입): VAD 경계확장이 클램프보다 먼저 적용되어, 실제 무음갭이 거의 없는(빠른 티키타카) 구간에서 다음화자 실제발화 시작까지 확장. **자막 타이밍도 VAD 말소리유무 이분법도 둘 다 이 케이스엔 무력** — 다음 시도는 음향 에너지/VAD확률 국소최소점에 직접 스냅.

**28화 규모 판단:** 개별 이론화보다 실측이 낫다고 판단 — 소량 꼬리누출(부분오염, 캐릭당 랜덤분산)이면 통계적으로 흡수될 가능성, 특정 콤비캐릭에 몰리면 그 캐릭만 선별정제. 페른(83% 전체오염)과는 질적으로 다른 문제(꼬리 0.1~0.3초 vs 클립 전체).

## 6. 28화 확장: 2단계 아키텍처

**문제:** 화별로 따로 클러스터링하면 화마다 번호가 무관 (1화 cluster_00 ≠ 2화 cluster_00).

**해결:**
- **Phase A** (`extract_episode.py`): 화별 추출(CTC+VAD스냅+하드클램프+임베딩), 클러스터링 없음. 출력: 클립+`embeddings.npz`+`metadata.csv`.
- **Phase B** (`identify_global.py`): 전체 화 풀링 → 자막태그 있는 클립으로 캐릭터별 centroid(전체통합, ≥3개 태그 요구) → 전체 클립을 최근접 centroid에 배정(임계값 미달시 `uncertain/`) → 태그-임베딩 불일치는 QA 플래그.

## 7. 클러스터 운영 이슈 (교훈)

- **자원공유 vs 데이터격리는 별개 축.** Slurm 파티션은 `AllowGroups=ALL`(공유풀), 홈디렉토리는 `drwx------`(완전격리) — CPU/RAM/GPU는 같이 쓰지만 파일은 안 보임. `squeue`로 전체 사용자의 잡(계정·자원량·경과시간)이 보이는 건 정상(공정배분 투명성 목적), 파일 내용과는 무관.
- **다른 팀 잡이 메모리를 크게 잡으면 GPU 자원요청도 막힘.** 3GPU 요청시 GPU당 최소메모리 정책(41000M)으로 인해 노드 전체메모리를 요구하게 되어, 다른 팀이 CPU/메모리만 82GB 써도 대기(Resources)에 걸림. `--mem` 명시해도 파티션 최소치가 우선 적용될 수 있음.
- **완전 idle 노드로 옮기는 게 실용적 해법.** V100-16(n13~17) 외에 **P40 노드(n8~11, 각 4장×24GB)** 발견. Pascal(SM6.1, V100의 SM7.0보다 구세대, 텐서코어 없음)이지만 cuDNN 비활성 우회가 아키텍처 무관하게 작동해 문제없이 실행됨.
- **워커 1개 실측 사용량**: VRAM ~1.67GB, RAM ~1.5GB. P40 4장×7워커=28워커로 **28화 전체를 한 번에 동시처리** 가능(GPU당 11.7GB/24GB, 전체 RAM 42GB/257GB, 여유 충분).
- **하드웨어 결함 발견**: 28화 동시처리 중 `ep19`가 GPU2에서 `CUDA error: uncorrectable ECC error encountered`로 실패 — 소프트웨어 버그 아닌 **실제 VRAM 하드웨어 결함 의심**. 다른 GPU로 재시도 필요, 관리자 보고 대상.

## 9. 텍스트정제 + v2Pro 재학습 착수, 캐릭터 선정 기준 확정 (2026-07-10)

**정제 파이프라인 실행**: ♪(110건 제외) + 통칭태그 제외(4개화 이상 분산 = 매번 다른사람, `子供`17화·`衛兵`8화·`商人`6화·`盗賊`6화·`２人`28화·`少女`28화 등 3,868건 제외) + 후리가나(→히라가나만 유지)·말줄임표(→「、」) 정제. 잔존 확인(말줄임표·반각괄호 0건, 정상 반영).

**MIN_CLIPS=20 기준 33명 확정**하여 v2Pro 풀파인튜닝(베이스 사전학습모델에서 새로 시작, 이전 체크포인트는 오염데이터 기반이라 폐기) 착수. n8(P40) 3장(GPU0/1/3, GPU2는 하드웨어결함으로 배제) 병렬.

### 9-1. 신규 캐릭터 QC 중 발견: 단일씬 편중 데이터의 생성불안정 (신규 이슈)
`オルデン`(72클립, 전부 15화 한 장면에서만 추출) 학습 후 추론 테스트 중 **생성 오디오가 대상텍스트 앞부분을 건너뛰고 중간부터 시작**하는 현상 발견. single-ref·multi-ref(5클립 융합) 둘 다 동일 증상.

**진단:**
- 로그 확인 결과 텍스트 프론트엔드 처리(`Processed text from the frontend`)는 **전체 문장이 정상 전달됨** — 텍스트 분할/파이프라인 버그 아님.
- 즉 문제는 **GPT모델의 자기회귀 생성 단계**에서 발생(SoVITS/음색이 아니라 GPT/발화순서 문제). multi-ref도 SoVITS쪽만 안정화하지 GPT 불안정은 못 고침 — single·multi 동일증상이 이를 뒷받침.
- **디코딩 파라미터(temperature 1.0→0.6, top_k 15→5) 낮춰도 출력 길이가 완전히 동일(5.34s)** — 랜덤샘플링 노이즈가 아니라 **구조적(결정적) 문제**로 추정.

**가설: 학습데이터가 클립수 기준(≥20)은 넘겨도 전부 한 화·한 씬에 국한되면, GPT가 그 장면의 좁은 운율/리듬 패턴에 편향 학습되어 교차언어(한국어) 새 텍스트에 대한 어텐션이 불안정해질 수 있음.**

**시사점**: 캐릭터 선정 기준을 "클립수" 하나로만 볼 게 아니라, **몇 개 화에 걸쳐 분산됐는지도 함께 고려**해야 할 수 있음(예: ≥2~3화 분산 요구). 다음 배치부터 반영 검토 대상.

### 9-2. 9-1 가설 정정: 단일씬 데이터 문제 아니었음, 특정 레퍼런스 클립 문제였음

9-1에서 세운 "단일씬 편중 데이터 → GPT 생성 전반 불안정" 가설을 **후속 테스트로 반증**함:
- **재확인**: multi-ref 테스트가 "5개 클립 융합인데도 동일 증상"이라 원인을 특정레퍼런스가 아니라고 판단했던 게 **틀림** — multi-ref도 GPT 프롬프트용 "메인 레퍼런스"는 동일한 문제클립(`ひと月前に 魔族との大きな戦いがあってな`)을 그대로 썼고, 나머지 4개는 SoVITS 음색융합에만 관여함(GPT 자기회귀 생성엔 영향 없음). 즉 진짜 원인변수(메인ref)를 안 바꾼 채 테스트해서 잘못된 결론 도출.
- **정정테스트**: ①원래ref+짧은문장("안녕하세요"만) → 정상 ②**다른 톤 레퍼런스**+원래 전체문장 → **정상**(길이도 5.34s→5.98s로 늘어나며 앞부분 복원됨).
- **결론: 오르덴 GPT모델 자체는 멀쩡함. 문제는 자동선택된 특정 레퍼런스 클립 하나였음** — 페른 사례(자동선택이 짧은/나쁜 클립을 집어 잘림 유발)와 동일 계열의 함정.
- **교훈**: ref 자동선택 로직("4~8초 중 가장 긴 것")이 길이만 보고 "생성안정성"까진 보장 못 함. 추후 캐릭당 레퍼런스 후보 여러 개를 자동 교차검증(예: 같은 텍스트로 자기 자신 재합성해보고 이상 없는지 체크)하는 절차 추가를 고려할 만함.

### 9-3. 용어 정리: "레퍼런스 선정 문제"

자동선택기가 나쁜 클립을 골라 생기는 문제군을 명명. **"단일/다중 레퍼런스"(SoVITS 음색 융합 방식)와는 다른 축**이니 혼동 주의.

**상위 카테고리: 레퍼런스 선정 문제 (Reference Selection Problem)**
| 유형 | 사례 | 증상 |
|---|---|---|
| 클립잘림형 | 페른(0.6초 잘림 사건) | ref가 너무 짧아 출력이 잘림 |
| 생성스킵형 | 오르덴 | ref 길이는 정상이나, 특정 클립 특성으로 GPT가 타겟텍스트 앞부분을 건너뜀 |

공통 원인: 자동선택기가 "길이(4~8초)"만 보고 고르지, "이 클립으로 생성했을 때 안정적인지"는 검증 안 함.

### 9-4. 레퍼런스 선정 문제 근본원인 규명 + 공식 가이드 확인

GPT-SoVITS 공식/커뮤니티 가이드(README, DeepWiki 퀵스타트) 확인 결과, 레퍼런스 오디오 선정 기준:
1. 길이 3~10초
2. 잡음없이 깨끗, 단일화자만(노래·웅얼거림 배제)
3. 발음 명료
4. **"자연스러운 대화체 유지, 단조롭거나 과도하게 극적인 낭독은 피할 것"**
5. 레퍼런스 텍스트가 실제 발화와 정확히 일치해야 함

**오르덴 "생성스킵형" 문제의 실제 원인 = 4번 위반으로 추정.** 문제클립("한 달 전 마족과의 큰 전투가 있었지")은 심각한 전황보고 서술로 **과도하게 극적인 낭독**에 해당. 정상이었던 클립("너의 전사로서의 역량은 훌륭하다")은 자연스러운 대화체. 공식 가이드에 정확히 부합하는 실패사례.

**추후 자동선택 로직 개선 방향**: 길이(3~10초) 기준만으로는 부족 — 극적/서술조 대사(전투·비장한 독백 등)를 휴리스틱하게 걸러내거나(예: 문장길이·특정 어휘 패턴), 캐릭당 후보 여러 개를 자기재합성 테스트로 교차검증하는 절차 고려.

### 9-5. 9-4 정정: "극적인 낭독" 가설 틀림 — 실제 원인은 경계누출(tail-bleed)

사용자가 직접 두 레퍼런스 클립을 들어본 결과 **녹음된 자막 텍스트와 실제 음성 내용이 다르다**는 지적. 원본 `ep15.srt`를 직접 대조한 결과:

```
문제클립: 871.579~876.083 ひと月前に 魔族との大きな戦いがあってな
          876.166~878.836 その時に ヴィルトが戦死した        ← 다음줄, 클립끝(876.166)=다음줄시작(876.166) 완전일치
정상클립: 1335.751~1339.504 お前の戦士としての力量は すばらしいものだ
          1340.005~1341.298 お前さえよければ…                ← 다음줄, 클립끝(1340.005)=다음줄시작(1340.005) 완전일치
```

**두 클립 모두 다음 대사가 경계없이 그대로 잘려 들어간(zero-margin tail-bleed) 오염 클립이었음.** 9-4의 "과도하게 극적인 낭독" 결론은 틀림 — 실제로는 8-1에서 이미 정량화했던 그 경계누출 문제(전체 9,388클립 중 10.2%)와 동일 계열. 다만 **둘 다 누출이 있는데 문제클립만 스킵 증상을 일으킨 이유는 특정되지 않음**(누출 내용·길이 차이로 추정) — 결과적으로 원인규명보다 **"경계누출 자체를 사전에 걸러내는 것"이 실질적 해결책**이라는 결론으로 9-6에서 이어짐.

### 9-6. 해결책 구현: 경계-클린 레퍼런스 자동선택기

8-1에서 만든 "다음/이전 자막 경계까지 여유(gap)" 계산 로직을 **레퍼런스 선정 단계의 필터**로 재사용.

**`scripts/pick_clean_ref.py`**: 캐릭터별 후보 클립(3~8초) 중 `gap_next`(클립끝→다음자막시작)·`gap_prev`(이전자막끝→클립시작)이 모두 안전마진(기본 0.3초) 이상인 것만 통과시키고, 그 중 가장 긴 걸 고름.

> **구현 중 버그 한 번 발견+수정**: 클립 경계가 자막 경계와 **정확히 일치**하는 경우(진짜 문제 케이스) `bisect_right`/`bisect_left`를 반대로 써서 그 경계를 건너뛰고 gap이 크게 계산되는 버그가 있었음 → 방향 수정(`bisect_left`로 next_start, `bisect_right`로 prev_end) 후 재검증, 오르덴 문제클립이 정상적으로 걸러짐을 확인.

**`scripts/scan_all_refs.py`로 전체 62명(생성태그 포함) 스캔**: "기존 방식(길이만 보고 최장 클립 선택)"이 골랐을 후보를 검사한 결과 **32명이 경계오염(gap=0.00~0.2s) 클립을 레퍼런스로 쓰고 있었음.** 오르덴은 우연히 증상이 겉으로 드러난 케이스일 뿐, **훨씬 광범위한 구조적 문제**였음이 확인됨 (아이젠·에-데루도 오염 레퍼런스였으나 우연히 증상 미발현).

**`scripts/gen_clean_ref_manifest.py`로 전체 캐릭터의 안전 레퍼런스 후보 매니페스트 생성**: 62명 중 44명 안전후보 확보. 18명은 없음(대부분 학습대상 33명에 포함 안 되는 통칭태그). **33명 학습대상 중 유일한 예외는 ファルシュ**(안전마진 0.3s 미달, 최선 후보 gap=0.17s로 타협).

**남은 작업**: 학습(job 2460) GPU 여유 확보되는 대로 오르덴을 새 클린레퍼런스(`ep15_ep15_01379687_01384428.wav`, gap_next=2.75s/gap_prev=3.23s)로 재검증, 이후 33명 전체 추론에 이 매니페스트를 기본 레퍼런스로 적용.

### 9-7. 완료 28명 배치 자기소개 추론 QC — 새 버그 발견 + 재시도-QA 도입

완료된 28명 전체(단일/다중레퍼런스, `안녕하세요, 장송의 프리렌의 OO입니다. 제 한국어가 자연스러운가요?`) 배치 추론 후 직접청취 QC 진행.

**인프라 이슈 (해결)**: 추론 스크립트가 학습 스크립트와 달리 `~/gsv_pyfix`(P40/V100처럼 SM<7.5 GPU에서 cuDNN 9.16 비호환이라 꺼주는 `sitecustomize.py` 패치)를 PYTHONPATH에 안 넣어서, GPU 완전히 비어있는 노드에서도 HuBERT conv1d 단계에서 `RuntimeError: GET was unable to find an engine`로 매번 실패. PYTHONPATH에 추가해서 해결 — 학습 job과 무관한 완전 별도 노드(n9)에서 문제없이 28명 전체 추론 성공.

**발견된 버그 1 (해결)**: `pick_clean_ref.py`가 클립의 `jp_text` 내용을 확인하지 않아 **♪(음악) 클립을 레퍼런스로 그대로 뽑는 문제** — シュトルツ 1순위 후보가 `～♪`였음, 그 결과 single/multi 둘 다 사실상 무음/웅얼거림. `build_training_lists.py`의 `MUSIC_RE` 필터를 `pick_clean_ref.py`에도 이식(빈텍스트/♪클립 제외).

**발견된 사실 2 (근본적)**: 경계-클린 레퍼런스를 써도 **일부 캐릭터가 여전히 문장 앞부분("안녕하세요, 장송의")을 스킵**하는 증상 발생(ハイター/ファルシュ/フェルン/ユーベル/リヒター/レルネン 등, single/multi 안 가리고 산발적). 동일 캐릭터·동일 레퍼런스·동일 문장을 **재실행만 했는데 스킵 없이 정상 생성됨**을 확인 → **GPT 자기회귀 샘플링 자체가 확률적(top_k=15, top_p=1.0, temperature=1.0)이라 같은 입력도 매 실행마다 다른 결과가 나올 수 있음**이 원인. 9-2에서 규명한 "특정 레퍼런스 클립이 원인"(결정론적 실패, 오르덴 사례)과는 다른 계열 — **레퍼런스가 깨끗해도 잔여 확률적 스킵 리스크는 항상 존재**.

**해결책: 생성 후 자동 QA + 재시도 루프 (`scripts/retry_qa_infer.py`)**
- 생성 직후 오디오의 시작 0.4초 RMS와 전체 RMS의 비율(`ratio`)을 계산, `ratio < 0.3`이면 "앞부분 스킵/무음"으로 판정하고 최대 6회까지 재생성
- 실제 검증: シュトルツ_single이 1차 시도에서 `ratio=0.003`(사실상 완전무음)으로 걸렸다가 2차 재시도에서 `ratio=0.447`로 정상 생성 — 재시도 루프가 실효성 있음을 확인
- 주의: 초기 버전은 절대 RMS 임계값(예 `0.01`)을 썼다가 오디오가 int16 스케일이라 사실상 항상 통과되는 버그가 있었음 → **반드시 시작구간/전체 비율 같은 상대값으로 판정**해야 함

**슈톨츠 별도 이슈(데이터 부족)**: GPT-SoVITS 자체가 레퍼런스 3~10초를 강제하는데, ♪필터 적용 후 슈톨츠(38클립)의 안전마진(0.3s) 통과 후보는 전부 3초 미만. margin을 0까지 낮춰도 3초 이상 후보가 `ep09_ep09_01029744_01032985.wav`(gap_prev=0.05s로 타협) 단 하나. 레르넨도 동일 계열로 3.9초짜리 대체 클립 수동 지정. **이런 캐릭터는 자동선택기만으론 못 고치고 개별 확인이 필요.**

**추가 검증**: `gen_clean_ref_manifest.py`가 `pick_clean_ref.py`의 `pick()`을 재사용하지 않고 후보수집 로직을 독자적으로 중복 구현해서, ♪필터 수정이 매니페스트 생성에는 반영이 안 되고 있었음(슈톨츠/レルネン이 여전히 `～♪`를 1순위로 잡음) → `pick()` 직접 호출하도록 리팩터링, 재생성 후 42/62명 안전후보 확보(음악/빈텍스트 0건) 확인. 유저가 직접 안 짚은 나머지 19명 원본 배치파일도 시작구간/전체RMS 비율로 전수조사 → 추가로 숨은 실패 없음(유일하게 걸린 건 유저가 이미 지적했던 `ハイター_single` 원본 그 자체).

### 9-8. 새 실패유형 발견: 조기절단형 (프리렌 본인 케이스)

33명 전원 학습완료 후 마지막 5명(フリーレン/ラント/リュグナー/ヴィアベル/頑固婆さん) 최초 추론 — `フリーレン_multi`가 dur=1.42s로 나머지 캐릭터들(5~9초대)과 확연히 다르게 짧음. ratio체크(0.4초 시작구간/전체RMS)는 통과했지만 **문장을 끝까지 안 읽고 중간에 생성이 멈추는 새로운 실패유형**("조기절단형") — "생성스킵형"(앞부분 누락)과는 반대쪽 실패.

**대응**: `retry_qa_infer.py`에 `MIN_DUR`(문장 길이 대비 최소 기대 오디오 길이, 기본 4.0s) 체크를 ratio체크와 함께 추가 — 둘 중 하나라도 기준 미달이면 재시도. 프리렌_multi는 1차 시도(dur=2.94s)에서 재시도 걸렸다가 2차(dur=7.78s, ratio=1.673)에서 정상 생성 확인.

**결론**: 생성스킵형(앞부분 무음)·조기절단형(문장 미완성) 둘 다 근본원인은 동일 — GPT 자기회귀 샘플링의 확률적 변동. **레퍼런스가 아무리 깨끗해도 사후 QA(시작비율 + 길이) + 재시도 없이는 완전한 안정성을 보장 못 함.**
